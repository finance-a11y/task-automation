<objective>
Two surgical resilience guards, plus the wiring that activates them:
1. HARD-02 (rate limits): a `createRetryingFetch` wrapper around the injected `FetchLike`
   that retries ClickUp 429 (honoring `Retry-After`) and 5xx / network rejects with
   exponential backoff + jitter, capped at ~3 attempts, throwing a typed
   `ClickUpRetryError` on exhaustion. A `sleep` (and `random`) is injected so the backoff
   is fully deterministic offline; live behavior under a real 429 is human-deferred.
2. HARD-01 (no silent failure): a best-effort `reportErrorToThread` helper and Spanish
   message constants, wired into the parse-failure + generic paths in `process.ts` and the
   create-failure path in `interactions.ts`.

Purpose: This phase hardens the existing Flow A/B code — it wraps and guards, it does NOT
refactor the working capture/confirm logic. No new npm packages (node builtins + the
already-injected clients only).
Output: `src/clickup/retry.ts`, `src/slack/report.ts`, and the wiring in `app.ts`,
`process.ts`, `interactions.ts`.

  <name>Task 1: createRetryingFetch — injectable-sleep 429/5xx backoff wrapper (HARD-02)</name>
  <files>src/clickup/retry.ts, src/clickup/retry.test.ts</files>
  <behavior>
    - 429 then 200: the wrapped fetch returns the 200 after exactly one retry; the injected sleep is called once.
    - 429 with `Retry-After: 2` header: the recorded sleep delay equals 2000ms (seconds → ms); without the header it uses exponential backoff base*2^n.
    - Backoff includes jitter driven by an injected `random` (default Math.random) so two attempts at the same step are not identical; with a fixed fake random the delay is exact and asserted.
    - 5xx (e.g. 503) is retried the same way; a 4xx other than 429 (e.g. 400) is NOT retried — returned immediately.
    - A rejected underlying fetch (network error) is retried up to the cap, then the last error is rethrown.
    - After `maxAttempts` (default 3) still-retryable responses → throws `ClickUpRetryError` carrying the final `status` (so HARD-01 can render `(<status>)`); the error is an `instanceof Error`.
    - The returned function satisfies FetchLike (ok/status/json/text pass through unchanged on success).
    - Total injected-sleep calls never exceed maxAttempts-1; tests run with zero real time.
  </behavior>
  <action>
    Create `createRetryingFetch(fetch: FetchLike, opts: { sleep: (ms: number) => Promise<void>;
    maxAttempts?: number; baseDelayMs?: number; random?: () => number }): FetchLike`. Defaults:
    maxAttempts 3, baseDelayMs 1000. On each attempt call the underlying fetch; treat status 429 or
    5xx (>=500) as retryable. For a retryable response on a non-final attempt, compute the delay:
    if a `Retry-After` header is present read it as integer seconds → ms (read defensively via an
    optional `headers?: { get(name: string): string | null }` view of the response; the minimal
    FetchLike has no headers, so guard with optional chaining), else `baseDelayMs * 2^attemptIndex`
    plus jitter `random() * baseDelayMs`. Await the injected `sleep(delay)`, then retry. On the final
    attempt with a still-retryable status, throw `new ClickUpRetryError(status)` (export a small
    Error subclass carrying `.status`). For underlying-fetch rejections, retry up to the cap then
    rethrow the original error. Non-retryable responses (incl. 2xx and other 4xx) are returned as-is.
    No new deps — pure TS. Inject sleep/random for determinism (CONTEXT > Testing: injected sleep/now).
    In retry.test.ts use a fake fetch (vi.fn returning queued {ok,status,json,text,headers:{get}}
    objects) and a fake sleep that pushes delays into an array — assert sequence + delays + final throw.
  </action>

  </verify>
  <done>429→Retry-After honored, 5xx+network retried, 400 not retried, exhaustion throws typed ClickUpRetryError, all delays driven by injected sleep — retry tests green.</done>
