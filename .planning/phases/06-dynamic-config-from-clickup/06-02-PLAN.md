<objective>
Build the config provider and switch the resolver to provider-injection. The provider exposes `getClientes()` / `getMembers()` that fetch live data (plan 01 client), cache it (plan 01 Redis helpers), and on failure fall back to last-good then the static maps — with the curated alias overlay merged on top of the live names. The resolver (`resolveCliente` / `resolveAssignees` / `resolveTask`) takes the resolved config as an optional injected argument, defaulting to the static maps so the existing tests pass unchanged.

Purpose: This is the heart of DYN-01/02/03/05 — the single point where "where does config come from" flips from hardcoded to live-with-fallback, while keeping the static maps as the safety net and the resolver pure/sync over already-resolved data.

Output: `src/config/provider.ts` + provider tests; resolver functions accepting injected config; resolver tests adapted (existing assertions preserved via static defaults).
</objective>
